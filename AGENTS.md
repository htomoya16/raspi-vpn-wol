# AGENTS.md

本リポジトリは「VPN接続済みの端末から、自宅LAN内PCへWOLを投げるためのWebダッシュボード」を提供する。
構成は **フロント静的（React/Vite） + バック動的（FastAPI） + DB（SQLite） + nginx** である。

---

## 目的 / スコープ

- VPN（WireGuard等）で自宅LANへ到達できることを前提に、WOL送信・起動状態確認・ログ閲覧をWeb UIで行う
- インターネットへ管理画面を直接公開しない（VPN内アクセス前提）
- 将来的に複数PC管理、簡易認証、ログ/操作履歴の拡張を想定する

非スコープ：
- VPN自体の構築手順（別記事/別管理）
- OS依存の高度な電源制御（ACPI連携等）

---

## 学習目的（このリポジトリで身につけたいこと）

本プロジェクトは単なるWOLツールではなく、以下の技術領域を**実装しながら学ぶ**ことを目的とする。

### ネットワーク
- VPN前提の設計（「公開面を最小化する」考え方）
- LAN内前提機能（WOL）の扱いと、境界（VPN/ルータ/NAT）の意識
- HTTP / リバースプロキシ（nginx）の基本

### Webフロントエンド（React/Vite）
- SPAの基本（状態管理、画面構成、API連携）
- 開発サーバ（HMR）と本番ビルド（dist生成）の違い
- APIクライアントの整理（`src/api/`へ集約）

### Webバックエンド（FastAPI）
- REST API設計（エンドポイント、I/F、エラー設計）
- 実行系（systemd、ログ、例外処理）
- 外部コマンド/ネットワーク処理（WOL送信、疎通確認）をサービス層で分離

### DB（SQLite）
- 最小のDB設計（targets/logs）
- 1ファイルDBの運用（バックアップ、肥大化対策）
- トランザクション/排他の基礎

### 運用（nginx/systemd/デプロイ）
- 静的配信と `/api` リバプロの分離
- サービス常駐（systemd）とログ確認
- 「PCでbuild → ラズパイへ配布」の実運用

## ディレクトリ構成

- `frontend/` : React + Vite（静的配信用ビルド成果物を生成）
- `backend/`  : FastAPI（WOL/Status/Logs API）
- `deploy/`   : ラズパイ配備用（nginx設定、systemdユニット、スクリプト）
- `README.md` : セットアップ概要と使い方（ユーザー向け）

推奨構成例：
- `frontend/dist/` は成果物であり、基本はGit管理しない（PCでbuild → ラズパイへ配布）

---

## 開発方針（重要）

- **フロントはPCで開発/ビルド**する。ラズパイ3でのnodeビルドは基本避ける
- **APIはFastAPIに集約**する（Next等の別サーバは立てない）
- **nginxは静的配信と /api のリバプロ**のみ担当する（ロジックを持たせない）
- **ログ/設定はSQLite**に保存し、肥大化対策（件数制限/ローテ）を前提に設計する
- **VPN前提**なので、公開インターフェースは最小にする（原則LAN/VPN内のみ）

---

## 学習の進め方（実務スタイル）

- フロントエンド実装は**初期状態から段階的に**進める（まとめて一気に作らない）
- 1タスクごとに「目的」「変更範囲」「完了条件」を先に決める
- 実装は小さな単位で行い、各ステップで動作確認してから次へ進む
- 学習者が理解できることを優先し、都度「なぜその実装にするか」を明文化する
- APIクライアント分離、コンポーネント分割などの設計判断は、実装タイミングで理由を添えて記録する
- コマンドは学習者が実行する前提とし、提示時は「そのコマンドが何をするか」「なぜ今必要か」を必ず説明する
- 複数コマンドを提示する場合は、各コマンドの期待結果（成功時に何が確認できるか）もセットで記載する

## API（最小セット）

- `POST /api/wol`
  - 指定targetへWOL送信
- `GET /api/status?target=<id>`
  - 起動状態を返す（疎通確認方法は実装に依存）
- `GET /api/logs?limit=<n>`
  - ログ取得（最新n件）

## ローカル開発手順

### Backend（FastAPI）
- 依存導入：`pip install -r requirements.txt` または `uv/poetry` 運用
- 起動：`uvicorn app.main:app --reload --host 127.0.0.1 --port 8000`
- 開発時DB：`backend/app/db/app.db`（SQLite）

### Frontend（Vite）
- （Vite初期化後）依存導入：`npm ci`
- （Vite初期化後）起動：`npm run dev`（例: http://localhost:5173）
- 開発時はVite proxyで `/api` を `http://127.0.0.1:8000` に転送する想定である

### 本番ビルド
- （Vite初期化後）`npm run build` → `frontend/dist/` を生成
- `dist/` をラズパイへ配布し、nginxが配信する

---

## ラズパイ本番運用（概要）

- nginx
  - `/` → 静的ファイル（例：`/var/www/wol/`）
  - `/api` → `127.0.0.1:8000`（FastAPI）へリバプロ
- FastAPI
  - systemdで常駐（例：`wol-api.service`）
- SQLite
  - 1ファイルで保持（バックアップ/ローテ方針を決める）

---

## セキュリティ前提

- 基本はVPN内アクセスを前提とする
- APIは副作用がある（WOL送信）ため、最低限の対策を検討する
  - 例：LAN/VPN内限定 + 簡易トークン + レート制限（必要なら）
- ログには個人情報を入れない（IP/UAは運用目的の最小限に留める）

---

## コーディング規約

- Python：型ヒントを付ける、I/O境界を関数で分離する、例外は握り潰さない
- FastAPI：ルータ分割（`api/`）、ドメインロジックは`services/`へ寄せる
- DB：最初はSQLite直でもよいが、テーブル定義とマイグレーション方針は明文化する
- Frontend：APIアクセスは`src/api/`へ集約し、UIコンポーネントは`components/`へ分離する

---

## ドキュメント運用（docs/）

- 本プロジェクトの仕様・設計・実装メモは `docs/` 配下に継続的に蓄積する
- 新規機能追加や仕様変更時は、実装変更と同じタイミングで関連ドキュメントも更新する
- 1トピック1ファイルを基本とし、必要に応じて `docs/frontend/` `docs/backend/` `docs/deploy/` などで整理する
- 各ドキュメントには最低限「目的」「変更内容」「運用時の注意点」を記載する
- `README.md` は入口として保ち、詳細は `docs/` の各ページへリンクする

## 変更時のチェックリスト

- [ ] `/api` のI/F変更はフロントと同時に更新したか
- [ ] nginxの `/api` リバプロ設定とFastAPIのlisten先が一致しているか
- [ ] 追加ログでDB肥大化のリスクがないか（limit/削除方針）
- [ ] VPN前提の境界が崩れていないか（誤って外部公開していないか）
- [ ] 実装変更に対応する `docs/` 更新（新規作成/追記）を行ったか
